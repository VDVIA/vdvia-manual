비전 — 무엇을, 왜

Vision & Rationale · 불변

AI 네이티브 조직은 더 이상 미래의 개념이 아닙니다. 회사의 모든 맥락을 AI가 읽을 수 있게 축적하고, 매일 자가 개선하며, 필요한 도구를 직접 만들어 쓰는 조직이 등장하고 있습니다. VDVIA는 이 방식을 실제로 구축합니다.

핵심은 암묵지(暗默知)의 자산화입니다. 개인의 머릿속·경험에만 존재하는 판단과 노하우를 AI와 함께 기록하여 검색·재사용 가능한 회사의 자산으로 전환합니다. 이 복리가 쌓일수록 생산성은 AI를 활용하지 않는 조직을 압도하게 됩니다.

1

운영 철학 — 95% 효율 우선

Operating Principle · 불변

100%의 정확성보다 95%의 효율을 우선합니다. AI에 최대한의 실행 권한을 부여하여 빠르고 많은 일을 처리합니다. 회의록·야간 배치 등 자동 산출물은 별도의 사전 승인 없이 GitHub·Notion에 즉시 반영하고, 처리 내역을 Slack에 요약 보고합니다.

AI와 구성원의 판단이 동기화되고 데이터가 축적될수록 정확도는 자연히 100%에 수렴합니다. 오류는 Git 이력으로 즉시 복구할 수 있으므로, 멈춰서 승인받는 비용이 사후 수정 비용보다 큽니다. 단 하나의 예외 — 저장 위치가 확실치 않을 때는 추측하지 않습니다(§5·§7의 _inbox 원칙).

4

지식 구조 — 폴더 트리 = 조직도 = 머릿속

Repository = Org Chart

조직(5팀 + 외주 대행 사업부 + 전사 공통)을 폴더 구조와 일치시킵니다. 각 폴더의 CLAUDE.md가 그 팀의 목적·정본 위치·라우팅 규칙을 안내하는 색인입니다.

vdvia/ ├─ 00_compass/ 전사 공통 — 정체성 · doctrine · 결정 로그(decisions/) · 관계 정본 ├─ 10_strategy/ 전략기획팀 (Marco) — OKR·시장 인텔·대시보드 ├─ 20_product/ 제품개발팀 (Marco) — 컨셉·처방·규제·생산 ├─ 30_growth/ 제품성장팀 (John 팀장 + 이솔) — 마케팅·디자인 ├─ 40_sales/ 해외영업팀 (Marco) — B2B·총판·수출 ├─ 50_operations/ 경영지원팀 (Marco) — 재무·법무·물류·정부지원 role 봇 ├─ 60_agency/ 외주 대행 사업부 (John 리딩 + 이솔 서포트) — 외부 클라이언트 · 레몬더마와 분리 └─ 99_log/ 세션 raw 기록 — {구성원}/sessions/ (john·marco·sol)
결정론적 라우팅 — 모든 세션은 여기서 시작
  1. 세션 첫 입력을 받으면 작업에 바로 들어가지 않고 선택지로 팀부터 확정한다.
  2. 그 팀 CLAUDE.md의 라우팅 지도를 따라 리프(최종 목적지)까지 반복한다.
  3. 상위 체인 CLAUDE.md(루트 → 00_compass → 팀 → 리프)를 로드하고 📍 목적지를 한 줄 보고한다.
  4. 그 세션의 모든 산출물·결정·업데이트는 목적지 폴더에 저장한다 — 최신화 지점 = 목적지.
실전 예시 — 사람이 "메타 광고 크리에이티브 검토해줘" 입력 → ① 선택지 제시(30_growth·10_strategy·20_product·40_sales·50_operations·60_agency·00_compass) → 30_growth 선택 → ② 30_growth/CLAUDE.md 지도 따라 하위 선택지(마케팅·디자인 → US 퍼포먼스 등) 반복 제시 → 리프 도달 → ③ 루트 → 00_compass → 30_growth → 리프 순서로 CLAUDE.md 로드 → ④ 📍 목적지: 30_growth/marketing/us/performance · 컨텍스트 로드 완료 한 줄 보고 후 작업 시작. 이후 이 세션의 모든 산출물은 자동으로 그 폴더행.
예외: "라우팅 건너뛰기" 또는 이미 하던 작업의 명시적 목적지(예: "어제 하던 세럼 입찰 계속") 제시 시 생략 · 여러 팀에 걸치면 00_compass.
지식의 경계 — 개인의 판단 스타일·협업 선호는 각자의 Claude 메모리(비공유), 회사가 알아야 할 지식은 반드시 레포의 해당 폴더로. 이 이동을 자동으로 수행하는 장치가 §7 암묵지 파이프라인입니다. 세부 카테고리는 실제 업무를 시작할 때 각 팀이 점진 생성합니다 — 처음부터 깊게 설계하지 않습니다.

구조 유지 시스템 — 지도 = 실물

2026-07-01 구조 표류 사건의 재발 방지 장치

7월 1일, 우리는 안내판 16개가 23일 동안 존재하지 않는 폴더(10_marketing·20_marco)를 가리키고 있었음을 발견했습니다. 폴더는 이사 갔는데 안내판이 안 따라와 3명의 머릿속이 갈라진 것. 이 섹션은 그 사건의 조항화입니다. 원칙은 하나 — 지도(문서)와 실물(폴더·자동화)은 항상 같아야 한다.

규칙구조 유지 5규칙 (00_compass/playbook.md에 동시 등재)
규칙실제 사건
① 폴더를 옮기면 그 자리에서 안내판도 함께 고친다. 이관 완료의 정의 = 안내판 갱신 + drift_check 신규 0건 + 자동화 경로 재지정, 3개 전부.재편 "완료" 선언 후 16개 문서가 23일간 옛 경로 안내
② 자동 재생성물은 기본 gitignore. 커밋 예외(이력이 자산인 데이터)는 폴더 .gitignore 주석에 이유 명시. 자동화 커밋은 add -A 금지 — 산출물 경로만 명시적으로.재생성물 커밋이 diff 노이즈·push 충돌 유발, 발행 소스를 지울 뻔한 사고 예방 사례
③ 만들기 전에 기존 세부 규칙을 먼저 읽는다. 라우팅의 상위 체인 CLAUDE.md 로드는 건너뛰지 않는다.하위 폴더의 세밀한 .gitignore를 상위에서 덮어쓸 뻔함
④ 문서 충돌을 발견하면 한쪽을 지우지 말고 둘 다 적어 결정자에게.지쉘 아마존 소유권 — 결정문 vs README 충돌 → 병기 후 미팅 결정
⑤ 새 자동화는 §8 인벤토리 등재 + heartbeat 등록 없이 "가동" 선언 금지.tiktok-affiliate가 3주간 아무도 모르게 휴면

모델·토큰 가드 — 공유 한도 안에서 일하기

Team Plan 5시간 사용량 한도는 전 구성원이 공유

Team Plan 사용량 한도는 3인이 함께 씁니다. 큰 작업 하나가 5시간 한도의 수십%를 태울 수 있어, 모델 선택과 작업 규모 판단을 의식적으로 합니다.

모델쓸 때
Haiku단순 분류·요약·태깅·반복 후처리 (자동화 끝단). 토큰 가장 적음.
Sonnet일반 작업·운영·코드 작성·분류·문서. 95% 작업의 sweet spot.
Opus전략 사고·복잡 디버깅·창작·신청서 작성·아키텍처 결정. 진짜 필요할 때만.
Fable 5Mythos급 최상위, 한도 소모 최대. 최고난도 전략·아키텍처·복잡 창작만 — 실행형 대규모 작업엔 쓰지 않고 Sonnet 전환 후 진행.

자동화 스크립트는 claude -p ... --model {haiku|sonnet|opus} 또는 환경변수 CLAUDE_MODEL로 명시. 인터랙티브는 사람이 /model로 즉석 전환.

가드대규모 작업 사전 스코핑

아래 중 하나라도 해당하면 실행 전 AskUserQuestion으로 스코핑한다 (해당 없으면 바로 진행):

  • 읽거나 수정할 파일 ~15개 이상
  • 서브에이전트·워크플로 fan-out 필요
  • 브라우저 왕복 10회 이상 예상 / 대량 데이터 전수 처리
옵션내용권장 모델
라이트핵심 파일만, 검증 생략, 서브에이전트 없음Sonnet
기본통상 진행, 서브에이전트 최소Sonnet (판단 지점만 상위 모델)
헤비fan-out + 적대 검증 포함상위 모델 필요 시에만
모델 자동 제안 규칙 — 메인 세션 모델은 Claude가 직접 못 바꿉니다(/model은 사람 몫). 작업이 실행·반복형인데 세션이 Fable/Opus면 스코핑 시 /model sonnet 전환을 함께 제안합니다. 서브에이전트는 기본 sonnet으로 스폰 — 모델 미지정은 세션 모델을 그대로 상속하므로, Fable 세션에서 서브에이전트를 그냥 fan-out하면 안 됩니다. 기계적 수집·후처리는 haiku, 전략 판단이 필요한 에이전트만 상위 모델. 자동화 스크립트는 위 표대로 --model 명시.
