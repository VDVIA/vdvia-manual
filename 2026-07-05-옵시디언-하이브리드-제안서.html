1

왜 이 제안인가 — 배경

현재 지식관리 구조의 한계 진단

현재 VDVIA의 모든 지식은 vdvia GitHub 레포(마크다운 폴더) 하나에 있고, 사람도 AI도 이걸 읽습니다. 근간 구조로는 잘 돌아가지만 사람이 편집·탐색하는 경험에는 다음 세 가지 한계가 있습니다:

  • 구조가 한눈에 안 보인다 — 폴더 깊이가 깊어 전체 관계를 그림으로 파악하기 어렵다.
  • 비개발 팀원 진입장벽 — Marco·Sol에게는 코드 에디터·터미널보다 친숙한 편집 환경이 필요하다.
  • 자동 동기화 감각 부재 — 편집할 때 "저장하면 곧바로 반영"되는 경험이 없다.

옵시디언은 이 세 가지를 정확히 채우는 도구입니다 — 그래프 뷰(시각화), 친숙한 편집기, 자동 동기화. 그래서 도입을 검토합니다.

2

핵심 판단 — "대체"가 아니라 "역할 분담"

전체 개편은 다운그레이드 — 동기화 성숙도 비교로 확인

먼저 오해 하나를 풀어야 합니다. 옵시디언과 GitHub는 경쟁 관계가 아닙니다 — 같은 마크다운 파일을 다루는 서로 다른 도구입니다. 옵시디언은 폴더를 여는 편집기·뷰어, GitHub는 그 폴더를 버전관리·동기화합니다. vdvia 레포는 이미 마크다운 폴더라서, 둘을 동시에 쓸 수 있습니다.

"전체 개편"은 오히려 후퇴입니다. 개인지식관리(PKM) 도구의 데이터 동기화 성숙도는 통상 ①로컬 → ②옵시디언 Sync → ③GitHub 푸시 → ④VPS 4단계로 구분됩니다. 이 구분 자체가 GitHub를 옵시디언 Sync보다 상위 단계로 규정합니다 — 옵시디언 Sync만으로 운영하는 것은 "매번 푸시하기 번거로워서" 택하는 임시방편에 가깝습니다. VDVIA는 이미 그 번거로움을 자동화로 해결해 3단계에 있습니다. 옵시디언 Sync 중심으로 개편하는 것은 2단계로 내려가는 것과 같습니다.

전체 개편 시 실제로 깨지는 것들:

잃는 것이유

버전 관리git의 변경 이력·되돌리기·비교. 옵시디언 Sync 버전 기록은 약하고 기간 제한
AI 네이티브 근간"레포 = AI가 읽는 단일 출처"가 정체성인데 그 뿌리가 흔들림

3

제안 — 하이브리드 구조 추천

같은 파일, 두 개의 창구

지식은 계속 vdvia 레포(마크다운) 하나에 둡니다. 그 위에 사람은 옵시디언으로, AI·자동화는 git으로 각자 편한 방식으로 접근합니다.

사람 창구
옵시디언 — 보기 · 편집 · 그래프 시각화 · (선택) 모바일. Marco·Sol이 친숙하게 문서 작업
▲ 같은 마크다운 파일을 위아래가 공유 ▼
동기화·근간
git / GitHub — 자동화 봇 · AI(Claude Code) · drift 감시 · 버전관리 · 대시보드 발행. 지금 그대로 유지
결과: 앞서 제시한 3가지 요건(시각화·팀 친숙·자동 동기화)을 전부 충족하면서, AI 네이티브 근간은 1도 건드리지 않습니다. 개편이 아니라 덧대기입니다.
VS Code는 필수가 아닙니다. 외부 사례("옵시디언+VS Code+클로드" 조합)에서 VS Code의 역할은 두 가지뿐입니다 — ① 클로드 코드를 띄울 자리, ② 코드 파일을 사람이 직접 뜯어보는 편집기. VDVIA는 ①을 클로드 코드 데스크톱 앱이 이미 하고 있고, ②(코드 수정)는 AI가 맡는 일이라 Marco·Sol은 설치할 필요가 없습니다. 클로드가 스크립트를 어떻게 고쳤는지 눈으로 보고 싶은 사람만 선택 설치.
4

자동 동기화 해법 — Obsidian Git 플러그인

이 제안의 기술적 핵심

유일한 실질 관건은 "사람이 옵시디언에서 편집한 걸 어떻게 git에 반영하나"입니다. 답은 Obsidian Git 플러그인입니다 — 옵시디언 안에서 몇 분마다 자동으로 git 커밋·푸시·풀을 해주는 무료 확장입니다.

사람이 느끼는 것실제로 벌어지는 것
옵시디언에서 그냥 편집·저장플러그인이 15분마다(설정값) 자동 커밋 + 푸시
"알아서 동기화되는 느낌"뒤에서 계속 git에 쌓임 → 봇·AI·버전관리·복구 전부 그대로 작동
다른 사람 변경이 자동 반영플러그인이 자동 풀 → 최신 상태 유지
옵시디언 유료 Sync는 쓰지 않습니다. git과 이중 동기화가 되어 같은 파일에 충돌이 납니다. 동기화는 git 한 경로로만 — Obsidian Git 플러그인이 그 다리입니다.

노션 경계 — 옵시디언이 가져갈 것 vs 노션이 지킬 것

노션을 없애는 게 아니라, 잘하는 일만 남긴다

노션이 지금 하는 일은 사실 두 종류입니다. 옵시디언은 그중 문서만 더 잘하고, 데이터베이스·협업은 노션이 이깁니다. 그래서 노션은 살려두되 역할을 정리합니다.

일어디가 맡나판정
문서·위키·회의록 (텍스트 기록)옵시디언 + git — 마크다운이라 AI가 바로 읽고 버전관리됨. 즉시 반영·폴더 인지옵시디언
캘린더·태스크·상태판 (달력뷰·담당·필터)노션 유지 — 데이터베이스는 노션의 킬러 기능. 옵시디언은 흉내만 나고 약함노션
실시간 공동편집·외부 공유 (우정님 등)노션 유지 — 옵시디언은 로컬 1인 기반노션
오히려 지금이 더 깨끗해집니다. 현재 매뉴얼 §6은 "노션 홈 = GitHub 미러"라 문서를 노션에 한 번 더 복사하는 애매함이 있습니다. 옵시디언이 문서 창구가 되면 그 노션 문서 미러가 불필요해지고, 노션은 캘린더·태스크·협업만 맡는 더 선명한 분업이 됩니다.
단, 회의록 이관은 서두르지 않습니다. 회의 공지봇이 매일 노션에 미팅록을 자동 생성하는 등 노션에 걸린 자동화가 있습니다. 회의록을 옵시디언으로 옮기려면 이 봇부터 손봐야 하므로, 파일럿에서 감을 잡은 뒤 별도로 실험합니다(로드맵 4단계).
5·2

3층 구조와 "1층 원본" — 진짜 얻고 싶은 것

지식 계층 구조의 공백 진단 + 용량 관리 원칙

지식 관리의 3층 구조(원본 → 증류 → 규칙) 중 VDVIA는 1층(원본 저장)이 약합니다. 이게 옵시디언 도입에서 실질적으로 챙기고 싶은 지점입니다.

층VDVIA 현재상태
1층 원본99_log(세션 raw) + voc(고객 목소리, 7/5 신설)뿐. 회의 원본·CS 원본 등은 안 쌓임약함
2층 증류playbook·decisions·notes·strategy-state (암묵지 파이프라인 산출)강함
3층 규칙CLAUDE.md 체인 (루트→팀→리프 라우팅)강함
중요 — 1층은 옵시디언 "기능"이 아닙니다. 원본 저장은 폴더 + 습관이고, 옵시디언은 그 폴더를 보기 좋게 훑는 창일 뿐입니다. 7/5에 만든 voc/가 1층의 첫 벽돌 — 앞으로 회의 원본·CS 원본으로 이 규율을 넓히는 게 실질 목표입니다(로드맵 3단계).
용량 — 텍스트는 걱정 0, 미디어는 걱정 크다. 텍스트 원본(리뷰·댓글·회의록)은 몇 년 쌓여도 수십 MB라 git이 감당합니다. 하지만 이미지·영상·큰 PDF는 git이 버전마다 통째 저장해 금방 부풀립니다 → 미디어는 권한 걸린 Drive, git엔 링크·텍스트만. 옵시디언은 이미지 붙여넣기를 vault(=레포)에 저장하므로, 파일럿 때 첨부를 무시 폴더/Drive로 빼는 설정을 잡습니다.

암묵지 파이프라인과의 관계 — 안 잃어버린다

옵시디언 작업도 매일 배치에 안전한가?

확인 결과, 야간 암묵지 러너는 대화(클로드코드 세션 + Slack)를 긁어 증류하고, 실행 전 git pull로 최신 상태를 받습니다. 그래서 두 가지가 나뉩니다:

  • 안 잃어버리기 — 옵시디언 편집은 Obsidian Git이 커밋·푸시 → 레포에 남고 러너도 그 상태를 봅니다. ✅
  • 암묵지 추출 — 러너는 대화를 증류합니다. 옵시디언에서 손으로 쓴 문서는 이미 완성된 지식이라 다시 증류할 필요가 없습니다(그게 정상).

즉 "옵시디언·노션·슬랙·세션 다 긁는다"는 방향은 맞지만, 옵시디언 작업은 git이 이미 보존하므로 러너가 중복으로 긁을 필요는 없습니다. (원하면 "오늘 바뀐 문서 요약"을 git log로 야간 리포트에 얹을 수 있음 — 선택.)

6·1

읽기 규율 — "안 읽고 일하는 AI" 문제

옵시디언만으로는 해결되지 않는 진짜 병목 + 3층 해법 (2026-07-07 추가)

최근 반복되는 문제: 클로드가 레포에 이미 있는 맥락·결정을 읽지 않은 채 일을 진행해버린다. 결과는 재작업 — 토큰과 시간의 최대 낭비원입니다. 정직한 답부터: 옵시디언 도입만으로는 이 문제가 해결되지 않습니다. 옵시디언은 사람 쪽 창구이고, 이것은 AI 쪽 읽기 규율의 문제이기 때문입니다. 원인은 세 가지입니다:

  • 말한 맥락은 세션과 함께 사라진다 — 채팅으로 준 지시·배경은 그 세션이 끝나면 다음 클로드에게 존재하지 않는 정보다. 문서로 승격된 것만 남는다.
  • "뭘 읽고 시작할지" 목록이 없거나 낡았다 — 클로드는 세션에서 루트 CLAUDE.md 정도만 자동으로 읽는다. 폴더별 필독 목록이 없으면 자기 판단으로 검색 몇 번 하고 바로 일을 시작한다.
  • 문서끼리 링크가 끊겨 있다 — A 문서를 읽어도, 전제가 되는 B 결정이 링크로 이어져 있지 않으면 B의 존재를 알 방법이 없다.

해법은 3층입니다. 셋 다 레포에 사는 규칙이라 개인 습관이 아니라 구조로 작동하고, 누가(사람·봇·새 세션) 열어도 똑같이 적용됩니다:

층내용누가 · 언제
① 기록중요한 지시·결정은 채팅에 두지 않고 그 자리에서 문서로. 세션 마감마다 암묵지 정리 트리거 습관화 — 이미 있는 파이프라인의 사용률을 올리는 것3인 · 매 세션
② 지도일 받는 리프 폴더마다 CLAUDE.md 표준 골격: 정체 1줄 + 필독 목록 5개 이하 + 하지 말 것(폐기 링크) + 산출물 저장 규칙. 핵심 문서엔 관련문서 링크·상태(정본/초안/폐기) frontmatter. 목록이 짧아야 지켜집니다Claude · 1회 정비 + 유지
③ 게이트루트 CLAUDE.md에 1줄 규칙 추가: "작업 시작 전 목적지 필독 목록을 읽고, 읽은 파일을 한 줄로 보고한 뒤 시작". 봇 프롬프트들에도 필독 목록 명시 감사(봇은 세션 라우팅을 안 타므로)Claude 1회 · 전 세션 적용
③의 "읽었다 보고 한 줄"이 전 구성원용 검수 장치입니다. 예: 📚 로드: brand.md · _status.md · changelog 최근 30줄 — 이 한 줄만 보고도 Marco·Sol이 "어, 브랜드 가이드를 안 읽었네" 하고 작업 전에 잡아챌 수 있습니다. AI 내부를 몰라도 검수가 됩니다.
옵시디언의 역할은 "발견 도구"입니다. 그래프 뷰에서 연결 0개인 고아 문서가 점으로 드러납니다 — ②지도 층의 끊긴 링크를 사람이 눈으로 발견하는 검수 화면이 그래프 뷰입니다. 옵시디언이 문제를 직접 고치진 않지만, 문제를 보이게 만듭니다.
토큰 계산도 같은 방향입니다. 필독 5개를 읽는 비용은 수천 토큰, 안 읽고 잘못 일한 뒤 재작업하는 비용은 수십만 토큰. "읽고 시작"이 항상 쌉니다.
